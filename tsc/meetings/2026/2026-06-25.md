# MoonRay TSC Meeting - June 25, 2026

## Quick Recap
The meeting focused on CI progress updates and versioning scheme discussions for MoonRay.   Randy also provided updates on DreamWorks' progress moving to ASWF.

On the CI front, J-F Panisset has made initial contributions to the ASWF-Docker repo with Conan recipes for MoonRay-specific dependencies like Embree and OpenImageDenoise (OIDN), and is testing builds for VFX Reference Platforms 2023-2025.

J-F is also working on pre-release images for refplat 2027, based on Rocky9 and Cuda 13 and should be done by end of week.  This will include moonray-ci images.  He also notes that we can pull pre-built Conan packages if we want.  Furthermore, he notes a couple of challenges with ISPC and OIDN due to hard-coded RPaths and TBB version conflicts, that are pulling pre-built binaries from Intel's GitHub.   He suggests we could try an older version of OIDN to see if that works.

On the versioning front, the team had a good discussion on various versioning schemes, with consensus forming around a date-based system using ISO format (year-week-calendar week) with decimal separators instead of the ISO standard letter, and potentially including patch versions.  E.g. YYYY.WW.X (Year, CW, Patch).  J-F suggested that we're diligent on formatting, for instance, choosing if we'll zero pad digits so that calendar weeks would be 01 and 52, for example.   It was also brought up to perhaps add a leading single digit that indicates the versioning system in use, which is similar to what Pixar does with USD.  E.g. X.YYY.WW.X (Versioning system, Year, CW, Patch).   This was not a hard requirement, and may be overkill in our use case.

The discussion also started exploring options for running regression tests (RATS) in the ASWF environment, noting the limitations of GitHub Actions free runners versus paid runners, and considering alternatives like AWS CodeBuild or third-party CI hardware providers.  

The conversation ended with a discussion about Jakub's contributions for Houdini integration, where Rob explained they would need to break the large pull request into smaller, more atomic changes, and we would definitely credit Jakub for his work on the Hydra 2 implementation.

## Summary
### OpenMoonRay CI Progress Updates:
Randy led the meeting in place of Jon who was on vacation. He provided updates on CI progress for OpenMoonRay, including initial contributions to the ASWF-Docker repo with Conan recipes for Moonray-specific dependencies and testing builds for VFX Reference Platforms. The team plans to release these builds to OpenMoonRay after further testing, including a Mac build, with GitHub actions to build a Moonray Docker container and unit/smoke tests, aiming for a go-live by the end of the month.

### RefPlat Container Image Updates:
Jean-Francois reported progress on releasing updated container images for RevPlat 2027, including CUDA 13 and Rocky9, with plans to complete the full set by the end of the weekend. He identified challenges with ISPC and OpenImageDenoise dependencies, particularly regarding libTBB version compatibility and hard-coded RPaths in Intel-provided binaries. The team confirmed that CUDA is the primary GPU API expected by Moonray, with CPU support as a secondary requirement. Randy announced a Birds of a Feather session scheduled for July 20th during Open Source Days, and introduced a proposal for a date-based versioning scheme using the ISO format for OpenMoonRay releases, moving away from the current, internally-divergent versioning scheme.

### Open Moonray Versioning Scheme Discussion:
The team discussed versioning schemes for Open Moonray, focusing on semantic versioning for shader APIs and scene files rather than the entire project. Paul explained that semantic versioning for the whole project would be impractical due to frequent changes, but versioning for shaders and scene files would be beneficial. Jean-Francois and Matthew recommended using a 4-digit year followed by a 2-digit week number with dot separators instead of the ISO standard's letter format, suggesting this would work better with existing packaging tools and be more consistent. The team agreed to present this proposal to Jon and potentially on Slack for feedback. The meeting then moved to discussing regression testing options for the Render Acceptance Test Suite (RATS), with Randy noting they currently run tests per machine class but need to determine how to recreate this in ASWF.

### GitHub Actions Regression Testing Challenges:
The team discussed challenges with running regression tests on GitHub Actions, particularly regarding CPU requirements and costs. Jean-Francois explained that free GitHub runners only provide 2 cores while paid runners offer 8-16 cores, with ASWF having a $1,500 monthly budget for paid runners. The group explored alternatives including AWS CodeBuild and third-party CI providers, with Paul noting concerns about compute costs for their testing needs. Toshi reported that some unit tests require 8 cores, though the team determined they might not need actual physical cores for testing thread affinity. The conversation ended with a discussion about Jakub's contributions to Houdini integration, where Rob indicated the large PR would need to be broken down into smaller, more atomic changes for easier merging, though the team would provide credit for Jakub's contributions in the upcoming Hydra 2 release.
